# Work ID

20260425-opencode-plugin-dep-b7c2

# Objective

Permanently fix Supercode package installation/runtime loading so OpenCode can load the Supercode plugin from its cache without requiring a manual `npm install @opencode-ai/plugin` workaround.

# Current State

- `package.json` imports `@opencode-ai/plugin` at runtime from Supercode source files, including `src/index.ts` and tool modules.
- `package.json` currently lists `@opencode-ai/plugin` only under `devDependencies` with version range `^1.3.10`.
- Runtime consumers that install Supercode without development dependencies can install the package successfully but fail when loading it because `@opencode-ai/plugin` is unavailable.
- User reproduced the failure as `Cannot find module '@opencode-ai/plugin'` during plugin load.
- User confirmed that manually installing `@opencode-ai/plugin@1.14.24` into the local OpenCode cache makes the plugin load and allows `opencode run "ping"` to respond normally.
- This repository uses `bun.lock` and no `package-lock.json`, `pnpm-lock.yaml`, or `yarn.lock` at repository root.

# Desired Outcome

- `@opencode-ai/plugin` is treated as a runtime dependency of Supercode, not only a development dependency.
- A cache/package install that omits development dependencies still includes `@opencode-ai/plugin`, allowing OpenCode to load the Supercode plugin without manual cache modification.
- Lockfile/package metadata remains consistent with the dependency change.

# Scope

In scope:

- Move `@opencode-ai/plugin` from `devDependencies` to `dependencies` in the root `package.json`.
- Use a runtime dependency version compatible with the user-verified working version, preferably `^1.14.24` unless repository constraints prove a different range is required.
- Update `bun.lock` or other package manager metadata generated by the repository’s package manager as needed.
- Verify package metadata and relevant tests/type checks sufficient to show the dependency is available for runtime packaging and existing TypeScript usage remains valid.

Out of scope:

- Changing OpenCode cache installation behavior.
- Adding a runtime fallback or lazy import workaround for missing `@opencode-ai/plugin`.
- Refactoring plugin/tool implementation beyond what is required for dependency metadata correctness.
- Publishing a release or modifying user-local OpenCode cache state.

# Non-Goals

- Do not redesign Supercode’s plugin architecture.
- Do not change the package name, exports, or plugin entrypoint unless required by verification failure.
- Do not introduce a different package manager or lockfile format.

# Constraints

- Keep the fix minimal and dependency-focused.
- Preserve existing project conventions: TypeScript package, Bun test runner, root `bun.lock`.
- The permanent fix must live in repository-tracked files rather than local cache state.
- No production implementation changes should be made unless verification proves they are necessary for this dependency fix.

# Success Criteria

- Root `package.json` lists `@opencode-ai/plugin` under `dependencies` and no longer lists it under `devDependencies`.
- The dependency version range is compatible with `@opencode-ai/plugin@1.14.24`.
- Repository lockfile/package metadata is updated consistently.
- Relevant verification commands pass, including at least dependency install/lockfile consistency verification and repository test/typecheck commands where available.
- The final change explains why `@opencode-ai/plugin` is a runtime dependency: Supercode imports it during plugin load/runtime, and OpenCode cache installs omit development dependencies.

# Risks / Unknowns

- The repository currently locks `@opencode-ai/plugin` to `1.14.20` through `bun.lock`; updating to `^1.14.24` may refresh transitive dependency metadata.
- If existing tests assume the older resolved version, they may need verification but should not require code changes.
- Full reproduction against OpenCode cache installation may depend on external OpenCode cache behavior and may be simulated or verified with the available local command if feasible.
